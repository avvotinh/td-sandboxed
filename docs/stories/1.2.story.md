## Story
**As a** hệ thống,
**I want** một ứng dụng Go có thể thiết lập và duy trì kết nối WebSocket ổn định tới TradingView,
**so that** có thể bắt đầu nhận dữ liệu thị trường real-time.

## Acceptance Criteria
1. Ứng dụng Go kết nối thành công tới endpoint WebSocket của TradingView.
2. Ứng dụng xử lý được quá trình xác thực (authentication).
3. Ứng dụng ghi log (logs) trạng thái kết nối (thành công, thất bại, mất kết nối).
4. Ứng dụng có cơ chế tự động kết nối lại nếu bị mất kết nối.

### Coding Standards
[Source: docs/architecture/phn-12-tiu-chun-code-coding-standards.md]

**CRITICAL - Must Follow:**
1. **NO direct environment variable access**: Use config package only
2. **NO hardcoded credentials**: Load from environment variables
3. **Structured logging ONLY**: Use `slog` package with JSON output
4. **Repository Pattern for database**: (Not applicable for Story 1.2, relevant for Story 1.4)
5. **Use context.Context in Go**: ALL I/O operations (WebSocket connect, read, write) must use context.Context

**Code Formatting:**
- Follow `go fmt` for all Go code
- Test files placed next to source files (e.g., config.go → config_test.go)

### Error Handling Strategy
[Source: docs/architecture/phn-11-chin-lc-x-l-li-error-handling-strategy.md]

**Logging:**
- Format: JSON
- Package: Use `slog` for Go
- Required log fields: timestamp, level, message, context data

**Retry Logic:**
- Ingestion client MUST have retry mechanism with exponential backoff when connecting to TradingView
- Recommended parameters:
  - Initial delay: 1 second
  - Max delay: 60 seconds
  - Backoff multiplier: 2x
  - Max retries: Unlimited (continuous retry with max delay cap)

**Idempotency:**
- Not applicable for Story 1.2 (relevant for batch processing in later stories)
