### Data Models
[Source: docs/architecture/phn-4-m-hnh-d-liu-data-models.md]

**TickData Model**:
- **Purpose**: Lưu trữ mọi biến động giá nhỏ nhất của một symbol
- **Attributes**:
  - `timestamp` (timestamptz): Thời gian chính xác của tick
  - `symbol` (text): Mã giao dịch (ví dụ: 'BTCUSD')
  - `bid` (decimal): Giá mua tốt nhất
  - `ask` (decimal): Giá bán tốt nhất
  - `price` (decimal): Giá khớp lệnh cuối cùng (có thể không có ở mọi tick)
  - `volume` (decimal): Khối lượng giao dịch của tick đó (nếu có)

### Redis Schema
[Source: docs/architecture/phn-8-s-database-database-schema.md]

**Key Pattern**: `latest_tick:{symbol}` (Example: `latest_tick:BTCUSD`)
**Data Type**: **Hash** with fields:
- `bid` (string representation of decimal)
- `ask` (string representation of decimal)
- `price` (string representation of decimal)
- `volume` (string representation of decimal)
- `timestamp` (RFC3339 formatted timestamp string)

**Example Redis Commands**:
```bash
# Write tick data
HSET latest_tick:BTCUSD bid "45000.50" ask "45001.00" price "45000.75" volume "1.5" timestamp "2025-10-02T10:30:00Z"

# Read latest tick
HGETALL latest_tick:BTCUSD
```

### Coding Standards
[Source: docs/architecture/phn-12-tiu-chun-code-coding-standards.md]

**CRITICAL - Must Follow:**
1. **NO direct environment variable access**: Use config package only
2. **NO hardcoded credentials**: Load Redis credentials from environment variables
3. **Structured logging ONLY**: Use `slog` package with JSON output
4. **Repository Pattern for database**: ALL Redis access MUST go through TickRepository interface
5. **Use context.Context in Go**: ALL I/O operations (Redis operations, WebSocket read) must use context.Context

**Code Formatting:**
- Follow `go fmt` for all Go code
- Test files placed next to source files (e.g., parser.go → parser_test.go)

### Error Handling Strategy
[Source: docs/architecture/phn-11-chin-lc-x-l-li-error-handling-strategy.md]

**Logging**:
- Format: JSON
- Package: Use `slog` for Go (already setup in Story 1.2)
- Required log fields: timestamp, level, message, context data
- New events for Story 1.3:
  - `message_received` - When WebSocket message arrives
  - `parse_success` - When message parsing succeeds
  - `parse_failed` - When message parsing fails (log error details)
  - `redis_save_success` - When tick saved to Redis successfully
  - `redis_save_failed` - When Redis save operation fails (log error details)

**Retry Logic**:
- For Story 1.3: No retry logic needed for Redis writes (fail fast, log error)
- Redis connection should be resilient via go-redis client's built-in retry mechanisms
