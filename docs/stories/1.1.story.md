## Story
**As a** người phát triển,
**I want** có một cấu trúc Monorepo chuẩn và một file Docker Compose để thiết lập môi trường,
**so that** có thể dễ dàng quản lý và chạy tất cả các thành phần của dự án một cách nhất quán.

## Acceptance Criteria
1. Một project Monorepo được khởi tạo.
2. File `docker-compose.yml` ở thư mục gốc được tạo, định nghĩa các service cơ bản: `ingestion-client`, `redis`, và `timescaledb`.
3. Có thể chạy lệnh `docker-compose up` để khởi tạo các service mà không bị lỗi.
4. Một file `README.md` cơ bản được tạo, hướng dẫn cách build và chạy môi trường.

[Source: docs/architecture/phn-9-cu-trc-th-mc-source-tree.md]

The monorepo structure must follow this exact layout:
```
hft-lakehouse/
├── apps/

├── data/
│   └── cold_storage/          # Chứa file Parquet
├── scripts/                   # Script hỗ trợ, benchmark
├── docs/
│   ├── prd.md
│   └── architecture.md
├── docker-compose.yml
├── .env.example
└── README.md
```

### Docker Compose Configuration Requirements
[Source: docs/architecture/phn-10-h-tng-v-trin-khai-infrastructure-and-deployment.md]
- **Deployment Model**: Docker Compose for MVP on self-hosted Linux server(s)
- **Deployment Strategy**: Manual deployment by pulling latest code and running `docker-compose up -d --build`
- **Rollback Strategy**: Revert to previous Git commit and redeploy

For Story 1.1, the docker-compose.yml must include:
- `redis` service (version 7.x)
- `timescaledb` service (PostgreSQL 16 + TimescaleDB 2.x)
- `ingestion-client` service placeholder (Go application - will be implemented in later stories)

### Database Schema
[Source: docs/architecture/phn-8-s-database-database-schema.md]

TimescaleDB will eventually need these tables (not required for Story 1.1, but good to know for initialization):
- `ticks` table with hypertable enabled
- `candles` table with hypertable enabled

Redis structure:
- Key pattern: `latest_tick:{symbol}` (e.g., `latest_tick:BTCUSD`)
- Data type: Hash containing fields: `bid`, `ask`, `price`, `volume`, `timestamp`

### Coding Standards
[Source: docs/architecture/phn-12-tiu-chun-code-coding-standards.md]

Critical Rules for all code:
1. Do not access environment variables directly
2. Do not hardcode credentials
3. Use only structured logging
4. Access database via Repository Pattern
5. Use `context.Context` in Go for I/O operations

Code formatting:
- Go: Follow `go fmt`
- Python: Follow `black`/`ruff`
- Test files should be placed next to source files

### Environment Configuration
[Source: docs/architecture/phn-12-tiu-chun-code-coding-standards.md]

All configuration must be loaded from environment variables, never hardcoded. The `.env.example` file should document all required variables without actual credentials.

### Infrastructure Notes
[Source: docs/architecture/phn-3-b-cng-ngh-tech-stack.md]
- Platform: Self-hosted
- MVP Deployment: Docker and Docker Compose on one or more Linux servers

### Testing

[Source: docs/architecture/phn-13-chin-lc-kim-th-test-strategy.md]

**Testing Approach**: Test-After for MVP, focusing on Unit Tests and Integration Tests

**Coverage Target**: >70% for new business logic

**Testing Infrastructure**:
- Integration tests will run on separate Docker Compose environment
- Test files should be placed adjacent to source code files

**For Story 1.1 Specifically**:
This story focuses on infrastructure setup, so testing will be:
- Manual verification that `docker-compose up` runs without errors
- Verification that all services (redis, timescaledb) start successfully
- No unit tests required for this infrastructure story

**Testing Framework** (for future reference):
- Python: Pytest 8.x
- Go: Standard testing package (future stories)
